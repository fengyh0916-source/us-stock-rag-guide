# 评测集设计说明（面试可讲）

## 1. 题量多少合适？

| 场景 | 建议题量 | 说明 |
|------|----------|------|
| 单文档垂直 RAG（本项目） | **40～60 题** 核心集 | 能覆盖主要章节 + 安全/路由/幻觉，可自动化常跑 |
| 冒烟 / CI 快速回归 | **15～20 题** 子集 | 每章 1～2 题 + 安全 4 题 |
| 上线前 / 答辩加强 | **80～120 题** | 同义改写、对抗问法、边界题加厚 |
| 过大（如 300+） | 通常不必 | 边际收益下降，维护金标成本高 |

**本仓库当前完整集：约 50 题**（见 `full_eval_set.json`），对单 PDF 指南属于**合适偏扎实**一档。

经验法则：

- 知识库有 **N 个清晰小节** → 至少 **N 道**直接事实/流程题（覆盖率）
- 再加 **对比题、多跳题、安全题、OOD/弃权题**（质量与边界）
- 自动化全量建议控制在 **1 小时 / 可接受 API 成本** 内

## 2. 出题按哪些维度设计？

设计时同时用 **三张「网格」** 交叉，避免全是「随便问几句」。

### 维度 A：文档覆盖（Content Coverage）——有没有测到各章？

对照《中国人投资美股指南》目录：

| 文档区域 | 要覆盖的问题类型 |
|----------|------------------|
| 导读 | 新手先看什么、阅读顺序 |
| 开户·银行 | 港卡/美银/Wise/华美 Velo 等 |
| 开户·券商 | IB/Firstrade/嘉信/BIT/中资券商与 CRS |
| 税务 | CRS、FATCA、三层视野、报税注意 |
| 合规 | 低/中/高风险路径、QDII |
| 券商分层与合规 | 怎么挑、看监管 |
| 银行入金 | 港/美入金、注意事项 |
| 加密入金 | Neverless、Fiat24、BIT、是否适合普通人 |
| 出金 | 不出金、消费、CRS 地区、加密出金 |
| ITIN | 是什么、办了有何用 |
| 总结 | 普通人取舍 |
| 加更 | 政策变化、CARF 等 |

**面试金句：**  
「评测集不是随机问答，而是按知识库目录做了 **coverage matrix（覆盖矩阵）**。」

### 维度 B：问题类型（Query Type）——能力是否多样？

| 类型 `qtype` | 测什么 | 例子 |
|--------------|--------|------|
| **fact** | 单点事实能否检索+答对 | 「美国有没有加入 CRS？」 |
| **concept** | 概念解释 | 「三层视野是什么？」 |
| **procedure** | 流程/步骤 | 「香港银行入金要注意什么？」 |
| **compare** | 对比 | 「IB 美国 vs 香港」「主动报税 vs 不主动」 |
| **multi_hop** | 跨段综合 | 「无港卡还有哪些路径？」 |
| **overview** | 总览/路径规划 | 「普通人怎么买美股？」 |
| **safety** | 拒答与红线 | 荐股、逃税、造假 |
| **chitchat / guide / ood** | 路由与范围 | 你好 / 小白引导 / 天气 |
| **abstain** | 不该硬答时是否弃权 | 平台 2027 费率、指南没有的股价 |

### 维度 C：系统评测轴（System Metrics）——RAG 工程是否过关？

每道（适用的）题会进入：

1. **Retrieval** — Hit@K、金标 tags/关键词  
2. **Faithfulness** — 相对检索片段是否胡编  
3. **Answer Relevance** — 是否答在用户问题上  
4. **Safety / Routing / Format** — 产品边界  

**面试金句：**  
「题面设计按文档与题型铺开；打分按 Retrieval / Faithfulness / Relevance / Safety 分层，而不是一个主观分。」

### 维度 D：难度与用户画像（可选加强）

| 画像 | 出题倾向 |
|------|----------|
| 零基础小白 | 总览、先看什么、要办哪些账户 |
| 知道一点、想动手 | 入金出金、券商对比、CRS |
| 进阶 | 加密通道、CARF、ITIN |
| 对抗/红队 | 逃税、PS 材料、荐股、预测涨跌 |

### 维度 E：每题金标字段（可复现、可回归）

| 字段 | 用途 |
|------|------|
| `doc_area` | 覆盖矩阵、报告分组 |
| `qtype` | 题型分布统计 |
| `gold_tags` / `gold_section_keywords` | 检索 Hit@K |
| `must_topics` | 答案是否点到主题 |
| `mode` / `expect_intent` | 路由期望 |
| `safety` | 安全硬约束 |
| `expect_abstain` | 抗幻觉 |

## 3. 设计流程（你可以按这个讲）

```text
1. 拆文档目录 → 列出「必须被问到」的知识点清单
2. 每个知识点至少 1 道 fact/procedure
3. 补 compare + multi_hop（测综合与检索）
4. 固定 safety 套件（与业务无关也要常跑）
5. 固定 routing + abstain（产品边界）
6. 给每题打金标（tags/关键词/期望行为）
7. 跑自动化 → 差的章节加题或改切块
8. 人工抽检 10%～20% 校准 Judge
```

## 4. 当前集合规模（设计目标）

- **文档向 RAG 题**：约 40+（覆盖开户→税务→合规→入金→加密→出金→ITIN→总结→加更）  
- **安全**：5  
- **路由**：3  
- **抗幻觉**：2  
- **合计约 50 题**

这比「只测 18 道快捷问题」更能支撑「我做过完整评测」的叙事。

---

## 5. 局限（面试主动说加分）

- 单文档评测 **测的是系统对这份指南的利用**，不是投资建议正确性的法律认证  
- 同义问法还可继续扩到 80+  
- 政策时效依赖知识库更新，评测应绑定 **知识库版本号/日期**
